# Phase 5 排程引擎規格（草案）

> 狀態：設計討論定稿前的工作版本。實作前若有調整，請直接更新本文件。
> 本文件是 Phase 5 的實作依據，schema 變更建議應在 Phase 3、4 實作時一併處理。

## 1. 目標

- 學生每天打開網站，立刻看到「今晚依序要做哪些任務」。
- 任務未完成、臨時新增小考、補習異動時，系統自動重排。
- 落後時優先「砍量保核心」，而不是把任務無限往後推。

## 2. 核心設計決策

| 決策 | 結論 | 理由 |
|---|---|---|
| 排程粒度 | 日級 + 時段帶（block），不做時刻表 | 國中生不會照時刻表執行；`StudyTask.plannedDate` 已是日級設計。時段帶補足「考前科目要排睡前」這類順序需求 |
| 引擎形態 | 純函數模組 `lib/scheduler/`，不碰 DB | input 為學生快照，output 為任務分配。可單元測試、可重跑 |
| 重排語意 | 每次從「今天」起全量重生成，不碰過去日期與 DONE/PARTIAL 任務 | 比增量搬移簡單且 deterministic：同樣 input 永遠產生同樣 output |
| 隨機性 | 引擎內禁止使用隨機數 | 保證可重現、可測試 |

## 3. 時段帶（Block）定義

一天切為粗時段帶，任務掛到「帶」而非精確時間：

| Block | 說明 | 適合任務 |
|---|---|---|
| `AFTER_SCHOOL` | 放學到家後～晚餐前 | 作業、輕複習 |
| `EVENING` | 晚上主時段（補習日 = 補習回家後，套用疲勞折扣） | 主要複習、練習 |
| `BEFORE_SLEEP` | 睡前約 30 分鐘 | 背誦類（國文、英文單字、社會）、明日小考最後複習 |

規則：

- 補習日 `fatigueLevel = HIGH` 時，`EVENING` 只排輕任務（背誦、訂正），不排新進度練習。
- 明日要考的科目優先佔用 `EVENING` 與 `BEFORE_SLEEP`。
- 學生看到的是「今晚順序清單」（block + sequence），不是時刻表。

## 4. 容量模型

```
每日可用分鐘 = 在家清醒時段
             − FixedEvent（晚餐、洗澡、家庭、自由時間）
             − TutoringSession（含 commuteMinutes）
             − 容量 buffer（預設 20%）
```

- 補習日疲勞折扣：`NORMAL` × 0.8、`HIGH` × 0.5（補習結束後的剩餘容量）。
- 容量計算以 weekday 循環為基礎，疊加日期例外（見 §8 DateOverride）。
- buffer 的目的：吸收估時誤差、保留自由時間，避免排滿導致放棄。

### 4.1 硬參數（來自 requirements.md 排程原則）

| 參數 | 值 |
|---|---|
| 每日自習量上限 | 國一 60–90 分、國二 90–120 分、國三 120–180 分。與作息推算容量取**較小值**作為當日上限 |
| 每日科目數 | 2–3 科。例外：明日多科小考的搶占情境（§7.1） |
| 單一任務長度 | 15–40 分鐘。超過 40 分鐘的任務必須拆分 |
| 休息 | 每 25–40 分鐘讀書安排 5–10 分鐘休息。休息**不生成任務**：由任務長度上限（40 分）與容量 buffer（20%）保證自然休息點，UI 在任務間提示休息即可 |

## 5. 任務池生成

| 來源 | 任務型態 | deadline 性質 |
|---|---|---|
| 學校作業 | `SCHOOL_HOMEWORK` | 硬 deadline（隔日或指定日） |
| 補習班作業 | `TUTORING_HOMEWORK` | 硬 deadline（下次上課前） |
| 小考（Quiz） | `EXAM_SPRINT` | 硬 deadline（考試前一晚） |
| 段考範圍 | `REVIEW` / `PRACTICE` | 軟 deadline，從考前往回鋪 |
| 間隔複習 | `REVIEW`（來自 ReviewItem） | dueDate |
| 弱點補強 | `WEAK_POINT` | 無 deadline，弱科加權 |
| 會考長期規劃（國三） | `REVIEW` / `PRACTICE` | 軟 deadline，獨立生成器 |

段考任務展開：`ExamScope` 章節 − 已完成章節 → 逐章展開為複習＋練習任務，從段考日往回平均鋪設，已教進度（ClassProgress）之後的章節標記為 `PREVIEW` 或暫不展開。

## 6. 排序與分配演算法

```
1. 計算規劃區間內每日每 block 的可用分鐘
2. 生成任務池（§5）
3. 排序：deadline 緊迫度 > priority > 科目交錯（避免單日全是同一科）
4. Greedy 裝箱：依序填入每日容量
   - 硬 deadline 任務必須排在 deadline 前
   - 軟 deadline 任務超載時往後推
5. 推不動時 → 觸發降級策略（§7）
```

輸出契約：

- 每日任務清單：科目、章節、類型、block、sequence、預估分鐘。
- 任務無法安排時**不得靜默丟棄**：回傳可理解的原因（容量不足／deadline 衝突／超出已教進度），由 UI 呈現給學生或家長。

## 7. 搶占與降級規則（核心）

多數讀書規劃工具失敗在「落後任務雪球式堆積」。本引擎明確規定：

### 7.1 搶占（明日多科小考）

1. 明日小考任務搶占一切軟 deadline 任務（段考／會考任務整批後移，不報錯）。
2. 當晚容量按科目加權分配，弱科多分。
3. 每科保底時間（預設 20 分鐘）：保證「每科都摸到」，而非一科讀完其他放棄。

### 7.2 降級（容量仍不足時，依序執行）

1. 砍 `PRACTICE` 保 `REVIEW`。
2. 縮小任務範圍（整章 → 重點／錯題），任務標題顯示縮減後範圍。
3. 砍每科時間至保底下限。
4. 永遠不砍科目數、不產生「今晚 0 分鐘」的科目（若明日要考）。

### 7.3 長期落後

- 軟 deadline 任務連續後移超過 N 天（預設 3）→ 標記該科進度警示，反映在進度追蹤頁。
- 段考剩餘天數 × 每日容量 < 剩餘任務總時數 → 自動切換「衝刺模式」：整批降級為重點複習，並通知學生／家長。

### 7.4 個別任務調整（承接 technical-plan 原規則）

- 任務連續未完成、或 `TaskLog.difficulty` 自評偏高 → 下次生成時拆小，並降級為基礎觀念任務。
- 未完成原因為「補習太累」→ 隔天只補短任務（≤ 20 分鐘），不整段回填當日缺口。

## 8. Schema 變更建議（Phase 3、4 一併處理）

| 變更 | 內容 | 理由 |
|---|---|---|
| `StudyTask` 加欄位 | `block`（enum）、`sequence`（Int） | 表達時段帶與當日順序，成本極低 |
| 新增 `Quiz` | `classroomId, subjectId, date, scope(String)` | 小考頻繁且班級共用，現有 `Exam` 偏段考粒度。班長／老師一人輸入全班受益 |
| 新增 `DateOverride` | `studentId, date, type(休假/段考週/補習取消), payload` | FixedEvent／Tutoring 只有 weekday 循環，無法表達假日、段考週、臨時請假 |
| 新增 `ClassProgress` | `classroomId, subjectId, chapterId, recordedAt` | 引擎需要知道哪些章節「已教完可複習」，對應 Phase 3「老師目前進度」 |
| `estimatedMinutes` 預設值表 | 以 `TaskType × 章節` 給預設值 | 初期無資料時的估時來源；後續用 `TaskLog.actualMinutes` 校正（schema 已預留） |
| （可選）`ScheduleRun` | 每次重排的快照與原因 | 回答家長「為什麼變這樣」；初期可先用 TaskLog.reason 替代 |

## 9. 測試案例（實作前先寫成單元測試）

引擎為純函數，以下案例均為「給定 input snapshot → 斷言 output 任務分配」。

| # | 情境 | 預期 |
|---|---|---|
| T1 | 無補習、無考試、僅段考任務池 | 任務平均鋪設，每日不超過容量 × 80% |
| T2 | 明日 1 科小考，容量充足 | 該科任務排入今晚 `EVENING`＋`BEFORE_SLEEP`，其他任務不動 |
| T3 | 明日 3 科小考＋補習日（HIGH）＋剩餘容量 60 分 | 3 科各 ≥ 20 分鐘保底；段考任務整批後移；補習後只排背誦／訂正 |
| T4 | 昨日任務 SKIPPED | 今日重排，未完成任務優先回填，過去日期不被修改 |
| T5 | 任務標記 DONE 後重跑引擎 | DONE 任務不被移動或重生成 |
| T6 | 容量不足且全為硬 deadline | 觸發降級鏈 §7.2，輸出含縮減標記；不丟任務 |
| T7 | 段考前 3 天，剩餘任務時數 > 剩餘容量 | 自動切換衝刺模式，全部降級為重點複習 |
| T8 | 同一日已有 5 科任務 | 科目交錯規則生效，新任務優先排他日 |
| T9 | DateOverride：週六全天家庭活動 | 該日容量為 0，任務繞過該日 |
| T10 | 補習取消（DateOverride） | 該日容量回升，任務可排入 |
| T11 | 背誦類科目（英文單字） | 優先落在 `BEFORE_SLEEP` |
| T12 | 相同 input 重跑兩次 | output 完全一致（deterministic） |
| T13 | 國三：會考長期任務與段考任務並存 | 段考前 2 週段考任務優先，會考任務降頻不清空 |
| T14 | 軟 deadline 任務連續後移 3 天 | 輸出進度警示 flag |

## 10. 開放問題

- 在家清醒時段的起點：用「到家時間」FixedEvent 推算，還是讓學生直接設定？
- 科目加權的初始值：純手動設定，還是用近期 Score 自動推算？
- 重排觸發時機：僅每日凌晨批次＋手動觸發，還是任何輸入變更即時觸發？

## 11. 提案：目標學校功能（待討論，建議列 Phase 8+ 或 backlog；mvp-tasks 的 Phase 7 為驗證階段，避免撞號）

### 11.1 功能描述

學生設定目標高中（如雄女），系統對照該校近年會考錄取門檻與學生目前成績，算出差距（gap），回饋到排程權重與會考長期規劃。

### 11.2 設計要點

| 層面 | 建議 |
|---|---|
| 資料來源 | 內建靜態資料表＋每年人工更新一次（每年 7 月放榜後）。高雄主要高中約 20 所，維護量極小。不建議爬蟲（來源不穩、維護成本高）或即時搜尋（不可靠） |
| 資料模型 | `TargetSchool`（學校基本資料）、`AdmissionRecord`（schoolId、年度、會考等級門檻、積分、序位區間、來源備註）、`StudentGoal`（studentId、schoolId、設定日期） |
| 回饋排程 | 目標門檻（如 5A）↔ 學生模擬考／段考等級 → 各科 gap → 調整 §7.1 科目加權與會考長期規劃的科目時間配比。gap 大的科目多排 `PRACTICE` 與 `WEAK_POINT` |

### 11.3 風險與注意事項

- 會考是等級制（A++～C）＋積分＋序位比序，每年浮動，不存在固定「分數線」。介面必須呈現為「歷年參考區間」，避免被當成保證門檻造成誤導。
- 對國中生顯示「你離雄女還差 XX」需要鼓勵性設計（顯示進步軌跡，而非單純差距），避免挫折感；建議家長端可見完整 gap 分析，學生端簡化。
- 價值高（目標驅動學習動機）、實作成本低（靜態資料＋權重調整），投資報酬率好，但依賴模擬考成績輸入（Score 已支援），國三才有完整資料。

### 11.4 「就差一點點」動機設計（goal gradient）

行為原則：人離目標越近投入越多（goal gradient effect）。設計核心是**永遠呈現「最近的可達成里程碑」的距離**，而非最終目標的距離。

#### 里程碑階梯（milestone ladder）

把「雄女」拆成由近到遠的階梯，介面永遠只強調下一階：

```
今晚任務全完成 → 本週連續達成 → 下次小考目標分 → 段考單科升一級（B++ → A）
→ 模擬考總等級 +1 → 目標學校門檻
```

顯示規則：

- 學生端主畫面只顯示「下一階」：「數學再對 3 題就升 A」「今晚完成 3 個任務，本週達成率就破 80%」。
- 距離一律用**學生可控的單位**呈現（題數、任務數、天數），不用等級差或抽象百分比。
- 進度條設計為「接近完成時視覺回饋加強」（最後 20% 顯眼化），利用 goal gradient 的視覺版本。
- 最終目標（雄女）只在「進步軌跡頁」出現，搭配歷史趨勢線：呈現「你正在接近」而非「你還差多少」。

#### 與排程引擎的連動

- 任務完成 → 即時回饋對該科里程碑的貢獻（「+1 題熟練度，距離 A 還剩 2 題」），讓每晚任務與長期目標有可見連結。
- 引擎在 gap 科目接近升級邊界時（如模擬考差 2 分升 A），短期內加排該科 `WEAK_POINT`／`PRACTICE`——在學生動機最高的時刻投放最有效的任務。
- 連續達成（streak）：中斷不歸零，提供每週 1 次「補救日」修復，避免 streak 斷裂造成放棄（loss aversion 反噬）。

#### 誠實底線（不可妥協）

- **不捏造接近感**：里程碑距離必須來自真實成績與任務資料。差距大時，把「接近感」放在微觀層（下一章熟練、下次小考），而非假裝最終目標很近。
- 不用倒數恐嚇（「會考剩 X 天你還差很多」），倒數只搭配可執行的下一步。
- 學生端永遠看得到「自己做得到的下一步」；完整 gap 分析放家長端。
